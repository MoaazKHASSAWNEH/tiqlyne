# Structifyx Motion Engine - Engine Events API

> Status: developer-facing reference.
> Scope: global engine events exposed through `createMotionEngine({ events })`.
> Goal: explain what the events are, when they fire, what payloads they receive, and how they should be used by applications, builders, logs and developer tooling.

## 1. Why engine events exist

The engine events API gives applications an observation layer around motion planning and playback.

It is designed for:

```txt
- debugging motion behavior
- logging motion lifecycle steps
- collecting analytics
- building visual builder inspectors
- displaying timeline diagnostics
- understanding generated execution plans
- tracking failures in production
- connecting future developer tools
```

The event system is not designed to mutate timelines or control playback.

Events are intentionally observational.

```txt
Events observe what the engine is doing.
Events must not become the primary way to change what the engine does.
```

This keeps the core predictable and avoids turning lifecycle callbacks into hidden business logic.

## 2. Public API

Events are configured globally when the engine is created.

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onBeforePlan(event) {
      console.log('Before planning', event);
    },

    onPlan(event) {
      console.log('Plan created', event.plan);
    },

    onPlay(event) {
      console.log('Playback started', event.target);
    },

    onFinish(event) {
      console.log('Playback finished', event.result);
    },

    onCancel(event) {
      console.log('Playback cancelled', event.result);
    },

    onError(event) {
      console.error('Motion error', event.error);
    }
  }
});
```

The first implementation is deliberately simple:

```txt
- no motion.on(...) yet
- no unsubscribe API yet
- no onProgress yet
- no per-frame events yet
- no step/track lifecycle events yet
```

This is a stable foundation. A dynamic subscription API can be added later without breaking the current `events` config object.

## 3. Event model

The event source is one of:

```ts
export type MotionEngineEventSource = 'registered-motion' | 'direct-timeline';
```

Meaning:

```txt
registered-motion
  The event comes from motion.play(target, config).
  The input is a MotionConfig that references a registered MotionDefinition by type.

direct-timeline
  The event comes from motion.playTimeline(target, timeline) or motion.planTimeline(timeline).
  The input is an explicit MotionTimelineDefinition.
```

Most lifecycle events share this base information:

```ts
export type MotionEngineEventBase<TTarget = unknown> = {
  readonly source: MotionEngineEventSource;
  readonly target?: TTarget;
  readonly motionId?: string;
  readonly motionType?: string;
  readonly timestamp: number;
};
```

Important notes:

```txt
target
  Present when the operation has a concrete playback target.
  It is required for play, finish and cancel events.

motionId
  Present for registered motion flows when the normalized config contains an id.

motionType
  Present for registered motion flows.
  Example: fade-in, slide-in, test-fade.

timestamp
  Generated by the engine with Date.now().
  It is useful for logs, debugging and timeline inspection.
```

Because the project uses strict TypeScript settings, optional fields must not be assigned as explicit `undefined`. If a field does not exist, the engine omits it from the event object.

## 4. Available events

The current event API exposes six callbacks:

```ts
export type MotionEngineEvents<TTarget = unknown> = {
  readonly onBeforePlan?: (event: MotionBeforePlanEvent<TTarget>) => void;
  readonly onPlan?: (event: MotionPlanEvent<TTarget>) => void;
  readonly onPlay?: (event: MotionPlayEvent<TTarget>) => void;
  readonly onFinish?: (event: MotionFinishEvent<TTarget>) => void;
  readonly onCancel?: (event: MotionCancelEvent<TTarget>) => void;
  readonly onError?: (event: MotionErrorEvent<TTarget>) => void;
};
```

These events cover the stable lifecycle of the engine:

```txt
before-plan
  Planning is about to start.

plan
  Planning completed successfully and produced a MotionExecutionPlan.

play
  Playback is about to be sent to the driver.

finish
  The driver completed playback and returned a MotionPlaybackResult.

cancel
  The cancel path returned a MotionPlaybackResult.

error
  Planning or playback failed inside play() or playTimeline().
```

## 5. Lifecycle order

### 5.1 Registered motion playback

Registered motion playback uses `motion.play(target, config)`.

Expected successful order:

```txt
onBeforePlan
onPlan
onPlay
onFinish
```

Flow:

```txt
motion.play(target, config)
  -> normalize config
  -> verify enabled flag
  -> resolve MotionDefinition from registry
  -> plan(config)
       -> onBeforePlan
       -> build timeline
       -> apply engine defaults
       -> validate timeline
       -> create MotionExecutionPlan
       -> onPlan
  -> onPlay
  -> driver.play(...)
  -> onFinish
  -> return MotionPlaybackResult
```

Example:

```ts
motion.register(new FadeInMotion());

await motion.play(element, {
  id: 'hero-fade-in',
  type: 'fade-in',
  options: {
    fromOpacity: 0,
    toOpacity: 1
  }
});
```

Event details:

```txt
onBeforePlan.source = registered-motion
onBeforePlan.motionId = hero-fade-in
onBeforePlan.motionType = fade-in

onPlan.source = registered-motion
onPlan.plan = MotionExecutionPlan

onPlay.target = element
onPlay.plan = MotionExecutionPlan

onFinish.target = element
onFinish.result = MotionPlaybackResult
```

### 5.2 Direct timeline playback

Direct timeline playback uses `motion.playTimeline(target, timeline, options?)`.

Expected successful order:

```txt
onBeforePlan
onPlan
onPlay
onFinish
```

Flow:

```txt
motion.playTimeline(target, timeline, options?)
  -> normalize play options
  -> planTimeline(timeline, options?)
       -> onBeforePlan
       -> apply engine defaults
       -> validate timeline
       -> validate reduced motion timeline when present
       -> create MotionExecutionPlan
       -> onPlan
  -> onPlay
  -> driver.play(...)
  -> onFinish
  -> return MotionPlaybackResult
```

Example:

```ts
const timeline = createMotionTimeline((timeline) => {
  timeline.track('self', (track) => {
    track.step((step) => {
      step.from({ opacity: 0 });
      step.to({ opacity: 1 });
    });
  });
});

await motion.playTimeline(element, timeline);
```

Event details:

```txt
onBeforePlan.source = direct-timeline
onBeforePlan.timeline = MotionTimelineDefinition

onPlan.source = direct-timeline
onPlan.plan = MotionExecutionPlan

onPlay.source = direct-timeline
onPlay.target = element
onPlay.plan = MotionExecutionPlan

onFinish.source = direct-timeline
onFinish.target = element
onFinish.result = MotionPlaybackResult
```

### 5.3 Planning without playback

Planning can be called directly:

```ts
const plan = motion.plan(config);
const directPlan = motion.planTimeline(timeline);
```

Successful registered motion planning emits:

```txt
onBeforePlan
onPlan
```

Successful direct timeline planning emits:

```txt
onBeforePlan
onPlan
```

Planning does not emit:

```txt
onPlay
onFinish
```

because no driver execution happens.

### 5.4 Cancel

Cancel uses:

```ts
await motion.cancel(target);
```

Current behavior:

```txt
- onCancel is emitted when the driver supports cancel.
- onCancel is also emitted when the driver does not support cancel.
```

If the driver does not support cancel, the result is:

```ts
{
  status: 'skipped',
  reason: 'driver-cancel-not-supported'
}
```

This is useful because logs and developer tools can see that a cancel attempt happened even when the driver cannot execute it.

## 6. Event reference

### 6.1 onBeforePlan

Type:

```ts
export type MotionBeforePlanEvent<TTarget = unknown> = MotionEngineEventBase<TTarget> & {
  readonly type: 'before-plan';
  readonly config?: MotionConfig;
  readonly timeline?: MotionTimelineDefinition;
};
```

Triggered when:

```txt
- motion.plan(config) starts planning a registered motion
- motion.planTimeline(timeline) starts planning a direct timeline
- motion.play(target, config) internally calls plan(config)
- motion.playTimeline(target, timeline) internally calls planTimeline(timeline)
```

Payload:

```txt
type
  Always before-plan.

source
  registered-motion or direct-timeline.

config
  Present for registered motion flows.

timeline
  Present for direct timeline flows.

motionId
  Present for registered motion flows when the normalized config has an id.

motionType
  Present for registered motion flows.

timestamp
  Date.now() value generated when the callback is emitted.
```

Use cases:

```txt
- log what is about to be planned
- inspect raw config/timeline input
- show visual builder activity
- start timing a planning operation
```

Example:

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onBeforePlan(event) {
      if (event.source === 'registered-motion') {
        console.log('Planning registered motion', event.motionType, event.motionId);
      }

      if (event.source === 'direct-timeline') {
        console.log('Planning direct timeline', event.timeline);
      }
    }
  }
});
```

### 6.2 onPlan

Type:

```ts
export type MotionPlanEvent<TTarget = unknown> = MotionEngineEventBase<TTarget> & {
  readonly type: 'plan';
  readonly plan: MotionExecutionPlan;
};
```

Triggered when:

```txt
- a registered motion plan is created successfully
- a direct timeline plan is created successfully
```

Not triggered when:

```txt
- validation fails
- the motion type is unknown
- the motion is disabled
- option validation fails
```

Payload:

```txt
plan
  The resolved MotionExecutionPlan produced by the engine.
  It contains the timeline that will be sent to the driver.
  It may also contain a reduced motion timeline.
```

Use cases:

```txt
- inspect the final resolved timeline
- debug defaults application
- display plan data in a visual builder
- collect analytics about animation structure
- generate documentation or previews from plans
```

Example:

```ts
events: {
  onPlan(event) {
    console.log(event.source, event.plan.timeline.tracks.length);
  }
}
```

### 6.3 onPlay

Type:

```ts
export type MotionPlayEvent<TTarget = unknown> = MotionEngineEventBase<TTarget> & {
  readonly type: 'play';
  readonly target: TTarget;
  readonly plan: MotionExecutionPlan;
};
```

Triggered when:

```txt
- the engine has a valid plan
- playback is about to be sent to the driver
```

Not triggered when:

```txt
- motion.play() returns skipped before planning
- planning fails
- validation fails
- unknown registered motion type is skipped
```

Important detail:

```txt
onPlay happens before driver.play(...).
```

This makes it useful for measuring driver execution time.

Example:

```ts
let startedAt = 0;

const motion = createMotionEngine({
  driver,
  events: {
    onPlay() {
      startedAt = Date.now();
    },
    onFinish(event) {
      console.log('Playback duration:', Date.now() - startedAt, event.result.status);
    }
  }
});
```

### 6.4 onFinish

Type:

```ts
export type MotionFinishEvent<TTarget = unknown> = MotionEngineEventBase<TTarget> & {
  readonly type: 'finish';
  readonly target: TTarget;
  readonly result: MotionPlaybackResult;
};
```

Triggered when:

```txt
- driver.play(...) resolves successfully
- the engine receives a MotionPlaybackResult from the driver
```

Important detail:

```txt
onFinish means the driver returned a result.
It does not necessarily mean the visual animation looked correct.
```

The result can contain statuses such as `finished`, `skipped` or other values supported by the driver/result model.

Example:

```ts
events: {
  onFinish(event) {
    console.log('Motion finished with status:', event.result.status);
  }
}
```

### 6.5 onCancel

Type:

```ts
export type MotionCancelEvent<TTarget = unknown> = {
  readonly type: 'cancel';
  readonly target: TTarget;
  readonly result: MotionPlaybackResult;
  readonly timestamp: number;
};
```

Triggered when:

```txt
- motion.cancel(target) returns from the driver
- motion.cancel(target) returns a skipped result because cancel is not supported
```

Current payload does not include `source`, `motionId` or `motionType` because cancel operates only on a target at the engine level.

Example:

```ts
events: {
  onCancel(event) {
    console.log('Cancel result:', event.result.status, event.result.reason);
  }
}
```

### 6.6 onError

Type:

```ts
export type MotionErrorEvent<TTarget = unknown> = MotionEngineEventBase<TTarget> & {
  readonly type: 'error';
  readonly error: unknown;
};
```

Triggered when:

```txt
- motion.play(target, config) catches a MotionPlanningError
- motion.play(target, config) catches an unexpected error
- motion.playTimeline(target, timeline) catches a MotionPlanningError
- motion.playTimeline(target, timeline) catches an unexpected error
```

Current behavior:

```txt
- onError is emitted before play() or playTimeline() returns a failed MotionPlaybackResult.
- onError is not emitted by plan() or planTimeline() when they are called directly and throw.
```

That distinction is important:

```ts
try {
  motion.planTimeline(invalidTimeline);
} catch (error) {
  // Direct planning throws to the caller.
  // It does not currently emit onError.
}

const result = await motion.playTimeline(target, invalidTimeline);
// playTimeline catches the planning error, emits onError,
// and returns a failed MotionPlaybackResult.
```

Use cases:

```txt
- production logging
- error reporting
- visual builder error panel
- analytics around failed animations
- diagnostics preview
```

Example:

```ts
events: {
  onError(event) {
    console.error('Motion failed:', event.source, event.motionType, event.error);
  }
}
```

## 7. Registered motion vs direct timeline payloads

### 7.1 Registered motion payloads

Registered motion events usually include:

```txt
source = registered-motion
motionType = normalizedConfig.type
motionId = normalizedConfig.id when available
config = original MotionConfig on before-plan
plan = MotionExecutionPlan on plan/play
result = MotionPlaybackResult on finish/error/cancel where relevant
```

Example:

```ts
await motion.play(element, {
  id: 'hero-intro',
  type: 'slide-in',
  options: {
    direction: 'bottom'
  }
});
```

Possible event values:

```txt
motionId = hero-intro
motionType = slide-in
source = registered-motion
```

### 7.2 Direct timeline payloads

Direct timeline events usually include:

```txt
source = direct-timeline
timeline = original MotionTimelineDefinition on before-plan
plan = MotionExecutionPlan on plan/play
result = MotionPlaybackResult on finish/error/cancel where relevant
```

Direct timelines do not currently have `motionType` or `motionId` because they are not registered definitions.

If an application needs an id for direct timelines, it should keep that id outside the core for now, for example in its own logging layer.

## 8. Skipped operations

Some operations return `skipped` before planning starts.

Examples:

```txt
- registered motion disabled
- registered motion type unknown
```

Current behavior:

```txt
motion.play() returns skipped for disabled configs before calling plan().
motion.play() returns skipped for unknown registered motion types before calling plan().
These early skipped returns do not currently emit lifecycle events.
```

This is intentional for the first version because those cases never enter the planning/playback lifecycle.

Potential future improvement:

```txt
Add onSkip(event) if skipped operations become important for analytics or visual builders.
```

## 9. Listener error behavior

The current implementation does not isolate listener errors.

That means if an event callback throws, the error can interrupt the operation.

Example:

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onPlay() {
      throw new Error('Listener failed');
    }
  }
});
```

This behavior is useful during development because listener bugs are visible immediately.

For production applications, event handlers should protect risky code:

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onFinish(event) {
      try {
        sendAnalytics(event);
      } catch (error) {
        console.error('Analytics failed', error);
      }
    }
  }
});
```

Future option:

```txt
eventErrorStrategy: 'throw' | 'ignore' | 'report'
```

This option is not implemented yet.

## 10. Events and exact optional property types

The project uses strict TypeScript behavior for optional properties.

Rule:

```txt
Do not set optional properties to explicit undefined.
```

Correct event shape:

```ts
{
  type: 'play',
  source: 'direct-timeline',
  target,
  plan,
  timestamp: Date.now()
}
```

Avoid:

```ts
{
  type: 'play',
  source: 'direct-timeline',
  target,
  motionId: undefined,
  motionType: undefined,
  plan,
  timestamp: Date.now()
}
```

This matters because event payloads are public typed objects. Consumers should be able to trust that optional properties are either present with a valid value or completely omitted.

## 11. Recommended logging patterns

### 11.1 Simple console debug

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onBeforePlan(event) {
      console.debug('[motion] before-plan', event.source, event.motionType);
    },
    onPlan(event) {
      console.debug('[motion] plan', event.plan.summary);
    },
    onPlay(event) {
      console.debug('[motion] play', event.target);
    },
    onFinish(event) {
      console.debug('[motion] finish', event.result.status);
    },
    onError(event) {
      console.error('[motion] error', event.error);
    }
  }
});
```

### 11.2 Analytics without breaking playback

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onFinish(event) {
      try {
        analytics.track('motion_finished', {
          source: event.source,
          motionType: event.motionType,
          status: event.result.status
        });
      } catch {
        // Analytics must not break UI playback.
      }
    }
  }
});
```

### 11.3 Visual builder inspection

```ts
const motion = createMotionEngine({
  driver,
  events: {
    onPlan(event) {
      builderInspector.showExecutionPlan(event.plan);
    },
    onError(event) {
      builderInspector.showError(event.error);
    }
  }
});
```

## 12. Current limitations

The current event API deliberately does not include:

```txt
- onProgress
- onPause
- onResume
- onRepeat
- onReverse
- onSeek
- onSkip
- onStepStart
- onStepFinish
- onTrackStart
- onTrackFinish
- onEveryFrame
- dynamic motion.on(...) subscriptions
- unsubscribe support
```

Reasons:

```txt
- progress events can be very frequent
- step/track events expose scheduler internals too early
- controller events need stronger cross-driver guarantees
- dynamic subscriptions need an unsubscribe contract
- scene/sequence APIs are not implemented yet
```

These events can be added later when the core scheduling model and driver contracts are ready.

## 13. Future event roadmap

Recommended future order:

```txt
1. onSkip
   For disabled motions, unknown motion types and unsupported driver operations.

2. onPause / onResume
   Once playback controllers have stable cross-driver semantics.

3. motion.on(type, listener)
   Dynamic event subscriptions with unsubscribe support.

4. onProgress
   Only with throttling or driver-level control.

5. step / track lifecycle events
   Only after timeline groups, scenes and scheduler semantics are more stable.
```

Potential dynamic API:

```ts
const unsubscribe = motion.on('finish', (event) => {
  console.log(event.result.status);
});

unsubscribe();
```

This is not implemented yet.

## 14. Stability rules

The following rules should be treated as part of the current event API contract:

```txt
- events are configured through createMotionEngine({ events })
- events are global engine listeners
- events are observational
- onBeforePlan is emitted before successful or failing planning attempts that enter plan()/planTimeline()
- onPlan is emitted only when an execution plan is created successfully
- onPlay is emitted before driver.play(...)
- onFinish is emitted after driver.play(...) resolves
- onCancel is emitted after cancel returns a MotionPlaybackResult
- onError is emitted by play()/playTimeline() when they catch planning or runtime errors
- plan()/planTimeline() throw errors directly and do not currently emit onError when called directly
- optional event properties must be omitted rather than set to undefined
```

## 15. Testing coverage

The current tests cover the main event behavior:

```txt
- planning events for direct timelines
- play and finish events for direct timelines
- complete registered motion playback event order
- error event when direct timeline playback planning fails
```

The expected successful registered motion order is tested as:

```txt
before-plan -> plan -> play -> finish
```

The expected successful direct timeline playback order is tested as:

```txt
play -> finish
```

Planning-only direct timeline behavior is tested as:

```txt
before-plan -> plan
```
